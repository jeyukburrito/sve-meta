# Shadowverse EVOLVE Agent Knowledge Pack

이 문서는 지금까지 대화에서 정리한 내용을 **압축 요약**한 문서다. 목적은 다음과 같다.

- 카드 DB + 룰북 기반의 개인용 RAG/하이브리드 에이전트 설계
- 카드 텍스트 조회를 넘어서 **룰 해석**, **아키타입 문맥 설명**, **초보적인 메타 해석**까지 가능한 구조 만들기
- 과한 통계 수집/실전 로그 없이도 현실적으로 운영 가능한 지식 구조 제시

---

## 1. 핵심 결론

### 1-1. JSON만으로도 시작은 가능하다
작은 개인 프로젝트에서는 카드 DB를 `json`으로 저장하고, Python 코드로 필터/검색한 뒤, 검색 결과 일부만 LLM에 전달해도 충분히 시작할 수 있다.

잘 맞는 질문 예시:
- 비용 2 이하 나이트메어 카드 찾기
- 특정 키워드/타입/클래스 카드 조회
- 카드명 exact/partial match

### 1-2. 하지만 JSON-only는 곧 한계가 온다
다음 질문은 JSON 필드 매칭만으로 답하기 어렵다.
- 왜 이 카드가 이 덱의 핵심인가?
- 이 효과는 언제 결정되는가?
- 이 카드 조합이 실제로 왜 강한가?
- 텍스트상 시너지는 있는데 왜 실전성은 낮은가?

이때는 카드 구조화 데이터 외에, **룰북 chunk 검색** + **아키타입 문서 검색**이 필요하다.

### 1-3. 최종 추천 구조는 하이브리드
가장 현실적인 구조:
- `cards.json` : 카드 사실 저장
- `rules_chunks.jsonl` : 룰북 조항/문단 chunk
- `archetypes.yaml` : 사람이 쓴 아키타입/플레이 문맥 문서
- `examples/*.md` : 좋은 설명 예시

즉,
- 카드 조회 → `cards.json`
- 룰 해석 → `rules_chunks.jsonl`
- 시너지/메타/플레이 설명 → `archetypes.yaml` + `examples`

## 2. 왜 룰북은 별도 RAG가 필요한가

Shadowverse EVOLVE 총합룰은 카드의 기본 구조(카드명, 클래스, 종류, 타입, 코스트, 카드 텍스트 등)를 명시적으로 정의하지만, 동시에 카드 텍스트 우선, 영역 이동, 진화, 선택, 키워드, 클래스별 특수 조건 같은 문단형 규칙이 많다.

에이전트가 다뤄야 할 대표 룰 포인트:
- 카드 텍스트가 총합룰과 충돌하면 카드 텍스트 우선
- 카드가 영역을 이동하면 원칙적으로 새 카드로 취급
- 묘지는 공개 영역
- 소멸은 묘지가 아니라 소멸영역으로 이동
- `Quick`는 상대 턴의 특정 타이밍 및 자기 턴 메인 페이즈에도 플레이 가능
- `Choice`는 플레이 단계에서 선택해야 함
- `Necrocharge`, `Spellchain`은 해석 시점 규칙이 있음

이런 정보는 단순 태그보다 **원문 근거 회수**가 중요하다.

---

## 3. 메타 해석에 필요한 것

### 3-1. 룰북만으로는 부족하다
룰북은 "무엇이 가능한가"는 알려주지만, "그래서 실제로 어떻게 굴리는가"는 알려주지 않는다.

예를 들어 룰북만으로는 부족한 질문:
- 왜 이 카드는 3장 거의 고정인가?
- 왜 텍스트상 좋아 보이지만 실제론 약한가?
- 왜 특정 매치업에서 mulligan/운영이 달라지는가?

### 3-2. 파인튜닝보다 먼저 필요한 것
아키타입을 이해시키기 위해 꼭 모델 파인튜닝이 필요한 것은 아니다. 현실적으로는 아래가 우선이다.

- 아키타입 정의 문서
- 핵심 카드와 그 이유
- 핵심 시너지 패키지
- 플레이 패턴
- 흔한 실수
- 설명 예시

즉, **암묵지의 최소 대체물**로 사람이 쓴 아키타입 문서와 QA 예시를 만든다.

---

## 4. 현실적인 범위에서 생략한 것

초기 단계에서는 아래는 생략 가능하다.
- 전 카드 global 역할 태깅
- 대회 decklist 대규모 수집
- 채용률/승률/매치업 통계 대규모 구축
- 플레이 로그/리플레이 자동 분석

대신 다음으로 대체한다.
- 핵심 카드 위주 local role 부여
- 아키타입 설명 문서
- 사람이 직접 작성한 좋은 설명 예시

## 5. 역할 분리 원칙

### cards.json
저장할 것:
- 카드명
- 클래스
- 종류
- 타입
- 코스트
- raw_text
- 키워드
- 관련 카드명

역할:
- 카드의 사실/원문 저장
- 필터/정렬/검색 기반 조회

### rules_chunks.jsonl
저장할 것:
- rule_id
- section path
- 원문 chunk
- 간단한 요약
- 검색 키워드

역할:
- 룰 근거 회수
- 타이밍/예외/상태 변경 설명

### archetypes.yaml
저장할 것:
- 아키타입 한 줄 정의
- 게임 플랜
- 핵심 카드와 이유
- 패키지 시너지
- 플레이 패턴
- 약점/실수

역할:
- 카드 단독 평가가 아니라 덱 문맥 설명
- 메타/시너지/플레이 감각 설명

### examples/*.md
저장할 것:
- 좋은 질문-답변 예시
- 답변 톤/판단 프레임

역할:
- 에이전트의 설명 스타일 보정
- 근거 없는 소설 방지

---

## 6. 카드 해석 원칙

카드에 대해 답변할 때는 항상 아래 순서를 우선한다.

1. **카드 사실**: 비용, 종류, 텍스트, 키워드
2. **룰 해석**: 타이밍, 영역, 처리 방식
3. **아키타입 문맥**: 왜 이 덱에서 채용하는가
4. **추측 여부 표시**: 근거가 없으면 추측이라고 표기

중요:
- 카드 단독으로 메타를 단정하지 않는다.
- 항상 아키타입 문맥에서 설명한다.
- 실전 데이터가 없으면 "운영 가설" 혹은 "텍스트/룰 기반 해석"이라고 밝힌다.

## 8. 한 줄 정리

이 프로젝트의 현실적인 목표는 **"모든 걸 아는 AI"가 아니라, 카드 DB와 룰북과 아키타입 문서를 근거로 설명하는 에이전트"**를 만드는 것이다.
